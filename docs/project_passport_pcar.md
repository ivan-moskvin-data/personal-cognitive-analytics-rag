# Personal Cognitive Analytics & AI Memory
*(Аналитика паттернов обучения и RAG-память для продуктового аналитика)*

## 1. Концепция продукта (Product Essence)

**Personal Cognitive Analytics & AI Memory** — это внутренний ИТ-продукт для продуктового аналитика, реализующий архитектуру RAG (Retrieval-Augmented Generation) для решения двух критических проблем:

1. **Knowledge Decay (Кривая забывания):** Потеря контекста при изучении SQL/Python из-за отсутствия систематического повторения и закрепления материала.
2. **Context Overflow:** Переполнение контекста в чатах с LLM при работе над долгосрочными проектами (например, Autonomous Stock Shadow), требующее постоянного копипаста архитектуры базы и прошлых решений.

Продукт представляет собой замкнутый цикл: **ETL-пайплайн → EDA-дашборд → RAG-агент**, где все данные извлекаются из личных логов общения с LLM (Google Takeout), векторизуются в локальную БД и используются для генерации персонализированных задач повторения и контекстно-зависимых ответов.

**Ключевые характеристики решения:**
* **Data-Centric Architecture:** Полная зависимость от собственных данных (чаты с Gemini), а не от внешних источников.
* **Active Recall Engine:** Агент работает как Coach, генерирующий микро-задачи на основе выявленных пробелов в знаниях.
* **Cloud-Inference, Local-Storage:** Тяжёлый инференс через API OpenRouter, хранение векторов и метаданных локально (ChromaDB).
* **Single-User Focus:** Продукт заточен под одного пользователя — продуктового аналитика, без мульти-тенантности и публичных интерфейсов.

## 2. Ключевой функционал (Core Features)

Система реализует три критических компонента:

* **ETL-пайплайн для чат-логов:** Парсинг сырых JSON-архивов из Google Takeout (логи общения с Gemini), очистка от служебных метаданных, нормализация структуры (timestamp, role, content, session_id). Экспорт в структурированный формат для последующей векторизации.
* **EDA-дашборд на Streamlit:** Исследовательский анализ паттернов обучения: распределение времени по темам (SQL vs Python), выявление сложных тем (частые вопросы, повторные запросы), трекинг прогресса по времени. Визуализация через `plotly`/`matplotlib`.
* **RAG-агент с памятью:** Векторизация очищенных логов в ChromaDB (локальная векторная БД). Подключение связки OpenClaw + Hermes Agent. Генерация ответов через API OpenRouter (модели Llama, Mistral, Claude). Агент извлекает релевантные фрагменты из истории для формирования контекста без галлюцинаций.

## 3. Проблематика и операционные барьеры (The Pain)

Текущий процесс обучения и разработки характеризуется критическими разрывами:

* **Кривая забывания (Ebbinghaus Curve):** Без систематического повторения 70% изученного материала теряется в течение 24 часов. Продуктовый аналитик тратит часы на повторное изучение уже пройденных тем SQL/Python.
* **Контекстное переполнение:** При работе над проектом Autonomous Stock Shadow каждый новый чат требует копипасты архитектуры БД, схемы таблиц и прошлых решений. Лимит контекста LLM быстро исчерпывается.
* **Отсутствие метрик обучения:** Нет способа измерить, какие темы вызывают сложности, где тратится больше всего времени, какие ошибки повторяются.
* **Ручной поиск в истории:** Поиск нужного куска кода или объяснения в логах чатов занимает минуты, а не секунды.
* **Галлюцинации при отсутствии контекста:** LLM генерирует ответы на основе общих знаний, а не на основе прошлых решений пользователя, что приводит к несогласованности архитектуры.

## 5. Продуктовая гипотеза (Product Hypothesis)

**Формулировка гипотезы:** Если мы внедрим RAG-систему на основе собственных чат-логов с LLM, то это позволит:

* **Сократить Time-to-Resolution:** Время на поиск нужного куска кода или объяснения сократится с минут ручного поиска до секунд RAG-запроса.
* **Повысить Context Retention:** Процент ответов агента, основанных строго на прошлых логах (без галлюцинаций), достигнет 90%+.
* **Увеличить Engagement:** Частота обращений к агенту для интервальных повторений составит 3-5 раз в неделю.
* **Обеспечить долгосрочную память:** Агент будет хранить контекст проекта Autonomous Stock Shadow без необходимости копипасты.

**Стратегическая цель:** Валидация гипотезы на этапе MVP с фокусом на ETL-пайплайн и базовый RAG-запрос.

## 6. Метрики успеха (KPI Framework)

Для оценки эффективности системы используются четыре группы показателей:

### Группа 1: Операционная эффективность (Operational Efficiency)
* **Time-to-Resolution:** Время на поиск нужного куска кода или объяснения в истории.
  * *Цель:* Сокращение с 5-10 минут (ручной поиск) до <10 секунд (RAG-запрос).
* **Search Accuracy:** Точность извлечения релевантных фрагментов из векторной БД.
  * *Целевой показатель:* >85% релевантных результатов в топ-3.

### Группа 2: Качество ответов (Answer Quality)
* **Context Retention (Доля успешных ответов):** Процент ответов агента, основанных строго на прошлых логах без галлюцинаций.
  * *Целевой показатель:* >90% для вопросов по проекту Autonomous Stock Shadow.
* **Hallucination Rate:** Процент ответов, содержащих информацию, не найденную в логах.
  * *Целевой показатель:* <10%.

### Группа 3: Вовлеченность (Engagement)
* **Active Recall Frequency:** Частота обращений к агенту для интервальных повторений.
  * *Целевой показатель:* 3-5 раз в неделю.
* **Task Completion Rate:** Процент сгенерированных микро-задач, выполненных пользователем.
  * *Целевой показатель:* >70%.

### Группа 4: Обучение (Learning Impact)
* **Knowledge Retention Rate:** Процент тем, которые пользователь не забывает через 7 дней после изучения.
  * *Целевой показатель:* >80% (против 30% без системы).
* **Time-to-Mastery:** Среднее время от первого контакта с темой до уверенного владения.
  * *Целевой показатель:* Сокращение на 40% за счёт Active Recall.

## 7. Архитектура системы и Technical Pipeline (3 этапа)

### Этап 1: ETL-пайплайн (Extract, Transform, Load)
**Задача:** Парсинг и очистка сырых JSON-архивов чатов из Google Takeout.

* **Input:** Сырые JSON-файлы из Google Takeout (логи общения с Gemini).
* **Transform:**
  * Парсинг JSON с валидацией структуры.
  * Очистка от служебных метаданных (timestamps в разных форматах, role-маркеры).
  * Нормализация: единый формат `{session_id, timestamp, role, content, topic_tags}`.
  * Обогащение: автоматическая теггация тем (SQL, Python, ETL) через lightweight NLP.
* **Load:** Экспорт в структурированный CSV/Parquet для последующей векторизации.
* **Технологии:** Python (`pandas`, `json`), `re` для парсинга.

### Этап 2: EDA-дашборд (Exploratory Data Analysis)
**Задача:** Построение дашборда на Streamlit для анализа паттернов обучения.

* **Metrics:**
  * Time Spent per Topic: распределение времени по темам (SQL vs Python).
  * Question Frequency: частота запросов по темам (выявление сложных тем).
  * Progress Tracking: трекинг прогресса по времени (линейный график).
  * Error Patterns: выявление повторяющихся ошибок (кластеризация).
* **Visualization:** `plotly` для интерактивных графиков, `matplotlib` для статических.
* **Технологии:** Streamlit, `pandas`, `plotly`, `seaborn`.

### Этап 3: RAG & Agent (Retrieval-Augmented Generation)
**Задача:** Векторизация логов и подключение агента.

* **Vectorization:**
  * Embedding: использование lightweight модели (например, `all-MiniLM-L6-v2`) для векторизации текстов.
  * Storage: ChromaDB (локальная векторная БД) для хранения векторов и метаданных.
  * Indexing: индексация по session_id, topic_tags, timestamp для фильтрации.
* **Agent Architecture:**
  * **OpenClaw:** Мультиканальный хаб для подключения Telegram/WhatsApp (опционально).
  * **Hermes Agent:** Мозг системы с памятью и инструментами (40+ встроенных инструментов).
  * **Inference:** Cloud API через OpenRouter (модели Llama, Mistral, Claude) — инференс не на локальном процессоре (Intel N150, 16GB RAM).
* **Retrieval Logic:**
  * Query: пользовательский запрос → embedding → поиск в ChromaDB.
  * Context Assembly: извлечение топ-K релевантных фрагментов + метаданные (session_id, timestamp).
  * Generation: промпт с контекстом → OpenRouter API → ответ.
* **Active Recall Engine:**
  * Detection: выявление тем с низкой частотой повторения (>7 дней).
  * Task Generation: генерация микро-задач на основе прошлых ошибок.
  * Scheduling: интервальное повторение по алгоритму SM-2 (SuperMemo-2).

## 8. Технологический стек (Technical Stack & Infrastructure)

**Программная среда (Software Stack):**
* **Python 3.10+:** Основной язык для ETL, EDA, RAG.
* **Streamlit:** Фреймворк для дашборда.
* **ChromaDB:** Локальная векторная БД для хранения эмбеддингов.
* **OpenClaw + Hermes Agent:** Агентные платформы для оркестрации.

**Облачные провайдеры (Cloud Providers):**
* **OpenRouter:** Единый API для доступа к моделям (Llama, Mistral, Claude).
* **Google Takeout:** Источник данных (логи чатов с Gemini).

**Аппаратное обеспечение (Infrastructure):**
* **Устройство:** Intel N150 (0.8 ГГц, до 3.6 ГГц Turbo), 16GB LPDDR5, 512GB SSD.
* **Операционная система:** Windows 11.
* **Обоснование:** Устройство не поддерживает локальные LLM, поэтому инференс полностью вынесен в облако через OpenRouter. Локально хранятся только векторы (ChromaDB) и метаданные.

## 12. Глоссарий (Glossary)

* **RAG (Retrieval-Augmented Generation):** Метод улучшения генерации LLM за счёт поиска релевантной информации во внешних источниках (в данном случае — в собственных чат-логах).
* **Active Recall:** Метод обучения, основанный на активном воспроизведении информации из памяти, а не пассивном повторении.
* **Knowledge Decay (Кривая забывания):** Психологический феномен, описывающий экспоненциальное забывание информации со временем без повторения.
* **Embedding:** Векторное представление текста, используемое для семантического поиска в векторных БД.
* **ChromaDB:** Легковесная локальная векторная база данных для хранения и поиска эмбеддингов.
* **OpenRouter:** Агрегатор API для доступа к множеству LLM через единый интерфейс.
* **ETL (Extract, Transform, Load):** Процесс извлечения, преобразования и загрузки данных.
* **EDA (Exploratory Data Analysis):** Исследовательский анализ данных для выявления паттернов и аномалий.
